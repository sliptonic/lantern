# Plan: Publishing Lantern to the TrueNAS Apps Catalog

**Status:** Research / not started
**Date:** 2026-06-02
**Prereq:** the GitHub repo is public.

## Goal

Make Lantern installable from inside TrueNAS via **Apps → Discover**, the same
one-click experience as Plex, Jellyfin, Actual Budget, etc. — instead of the
current "paste a compose file as a Custom App" path (already documented in
[`docs/truenas.md`](../docs/truenas.md) and [`deploy/truenas/`](../deploy/truenas/)).

## Background: how the catalog works now (Electric Eel / Fangtooth)

TrueNAS SCALE replaced its old Helm/k3s app engine with a **Docker + Docker
Compose** engine starting in **24.10 "Electric Eel"** (refined in 25.04
"Fangtooth"). The old `truenas/charts` repo and TrueCharts are legacy. The
current official catalog lives in one repo:

- **Repo:** https://github.com/truenas/apps
- **Market / docs:** https://apps.truenas.com
- **Contributor guide:** https://github.com/truenas/apps/blob/master/CONTRIBUTIONS.md
- **How to contribute (overview):** https://apps.truenas.com/getting-started/contributing-to-apps/

Apps are organized into **trains**: `charts` (legacy k3s migrations),
`enterprise` (iX-managed), and `community` (the one we submit to — open to
contributions; other trains are iX-only). An app is **not** built from a
Dockerfile on the user's box; the catalog references a **pre-built image** and
generates a `docker-compose.yaml` from a Jinja2 template plus a UI form
(`questions.yaml`).

## The two hard prerequisites (must do before any catalog PR)

### 1. Publish a pre-built, multi-arch image to a public registry

The catalog cannot `build:` from our Dockerfile — it pulls an image by
`repository:tag`. Today we only build locally (`image: lantern:latest`).

**Work:**
- Add a GitHub Actions workflow (none exist yet — `.github/workflows/` is empty)
  to build and push on tag/release to **GHCR** (`ghcr.io/sliptonic/lantern`).
- Build **multi-arch** (`linux/amd64` + `linux/arm64`) via `docker buildx` —
  TrueNAS runs on both; many home boxes are amd64 but arm64 is worth it.
- Adopt **semver tags** (`v0.1.0`, `0.1`, `latest`). The catalog pins an exact
  `app_version` (image tag), so every release needs an immutable tag.
- Note: Playwright/Chromium pins make the image large (~1.5 GB). Acceptable, but
  worth a layer-size pass before the first public tag.

### 2. Make the container run as a non-root user (uid/gid 568)

TrueNAS apps strongly prefer running as the host `apps` user (**uid/gid 568**),
and the catalog template runs a `permissions` init-container that `chown`s the
data dataset to the chosen uid/gid. Lantern's image currently runs as **root**.
Two specific risks to resolve and test:

- **Playwright browser path.** `playwright install` as root installs Chromium
  under `/root/.cache/ms-playwright`, unreadable by uid 568. Fix: set
  `ENV PLAYWRIGHT_BROWSERS_PATH=/ms-playwright` and `playwright install` into a
  world-readable path in the Dockerfile, or install as the target user.
- **Chromium sandbox.** Already launched with `--no-sandbox`
  ([`app/pdf.py:46`](../app/pdf.py)), which is what lets Chromium run unprivileged
  — good. Chromium also needs a writable `$HOME`/tmp for its profile; confirm
  `/tmp` is writable by 568 (it is by default).
- **`/data` ownership.** App writes a git repo, SQLite db, images, logo under
  `/data`. The perms container handles initial ownership; just confirm the app
  tolerates a pre-chowned, possibly non-empty dir.

**Decision to make:** bake `USER 568` into the image, or keep the image
root-capable and let the compose template set `user:`. The TrueNAS convention is
a configurable `run_as` (defaulting 568) set via compose, so keep the image able
to run as an arbitrary uid (don't hardcode `USER`), and make Chromium's browser
path + any cache dirs world-readable/writable. **Validate with a real run as
`--user 568:568` locally before submitting.**

## App file layout (target: `ix-dev/community/lantern/`)

Modeled on `ix-dev/community/actual-budget` (a near-twin: single container, one
web port, one `/data` volume). Files we author:

```
ix-dev/community/lantern/
├── app.yaml                     # metadata: version, image, categories, icon, screenshots
├── ix_values.yaml               # static defaults: image repo/tag, consts, container names
├── questions.yaml               # the install-time UI form
├── README.md                    # short description shown in the catalog
└── templates/
    ├── docker-compose.yaml       # Jinja2 template using ix_lib.base.render
    └── test_values/
        └── basic-values.yaml     # CI render/deploy test config
```

Auto-generated by CI (do **not** hand-edit / commit): `item.yaml`,
`templates/library/`, and the `lib_version_hash` field in `app.yaml`. PRs may
only touch files under `ix-dev/` and `library/`.

### `ix_values.yaml` (sketch)

```yaml
images:
  image:
    repository: ghcr.io/sliptonic/lantern
    tag: 0.1.0
consts:
  lantern_container_name: lantern
  perms_container_name: permissions
  base_data_path: /data
```

### `questions.yaml` — the install form

Reuse actual-budget's standard blocks verbatim for the boilerplate (Network /
Storage / Resources / Labels / `run_as`), and add a **Lantern Configuration**
group for our env vars:

| Field | Type | Notes |
|---|---|---|
| `base_url` | string, **required** | The LAN address phones will scan, e.g. `http://192.168.1.50:31100`. **Must match the published port chosen below.** This is the biggest UX gotcha — see Risks. |
| `edit_pin` | string, **private** | Space PIN. Default `changeme`; we surface the same default-PIN warning the app already logs. |
| `pin_enabled` | boolean, default true | Set false to run fully open. |
| `page_size` | string enum (`letter`/`a4`), default `letter` | |
| `seed_samples` | boolean, default true | |
| `additional_envs` | list | escape hatch (mirrors actual-budget) |

- **Web port:** internal container port stays **8080**; the published host port
  is user-configurable (catalog convention defaults to a high port like
  `31100`). The template maps `<published>:8080`.
- **Storage:** one entry, `data`, mounted at `/data`, defaulting to an
  **ixVolume** (auto-created dataset) with a **Host Path** option so existing
  users can point at their current `tank/apps/lantern` dataset and keep history.

### `templates/docker-compose.yaml` — render template

Mirror actual-budget's template:
- one container from `"image"`, `set_user(run_as.user, run_as.group)`;
- `healthcheck.set_custom_test` hitting **`/healthz`** (already exists,
  [`app/main.py:52`](../app/main.py)) — e.g. `curl -f http://localhost:8080/healthz`;
- env: `BASE_URL`, `EDIT_PIN`, `PIN_ENABLED`, `PAGE_SIZE`, `SEED_SAMPLES`,
  `DATA_DIR=/data`, plus `add_user_envs(additional_envs)`;
- `add_port(web_port)` (container side 8080);
- `add_storage("/data", storage.data)` + a `permissions` init-container that
  chowns it to `run_as`;
- `tpl.portals.add(web_port, {"scheme": "http"})` for the "Web UI" button.

### `templates/test_values/basic-values.yaml`

Minimal config so CI can render + boot the container and wait for `/healthz`:
set `base_url`, `edit_pin`, `pin_enabled: false` (so the boot doesn't warn),
`web_port.port_number`, `run_as 568/568`, and `storage.data` as an
`ix_volume`. Model on actual-budget's `basic-values.yaml`.

## Local development & validation

Per `CONTRIBUTIONS.md`:

```bash
git clone https://github.com/truenas/apps.git && cd apps
devbox shell            # or install: git, docker, docker-compose, python3, jq
                        # + pip: pyyaml psutil pytest pytest-cov bcrypt pydantic

# render only (no deploy):
./.github/scripts/ci.py --app lantern --train community \
  --test-file basic-values.yaml --render-only=true
# build + deploy + health-wait:
./.github/scripts/ci.py --app lantern --train community \
  --test-file basic-values.yaml
# keep running for manual poking:
./.github/scripts/ci.py --app lantern --train community \
  --test-file basic-values.yaml --wait=true

# devbox shortcuts:
devbox run app-render community lantern basic-values.yaml
devbox run app-test   community lantern basic-values.yaml

# metadata + port checks:
./.github/scripts/generate_metadata.py --app lantern --train community
./.github/scripts/port_validation.py
```

The CI script auto-generates `item.yaml`, copies the library into
`templates/library/`, sets `lib_version_hash`, renders the template, deploys,
and waits for health. Pre-PR checklist (from the guide): app boots with every
test file, `app.yaml` complete, `questions.yaml` labels clear, README written,
only `ix-dev/`/`library/` touched, no auto-generated files committed.

## Submission flow

1. Push the public image tag (prereq 1) so `app_version` points at a real,
   pullable image.
2. Fork `truenas/apps`, add `ix-dev/community/lantern/`, validate locally.
3. Open a PR against the **community** train. Use the
   [discussions page](https://github.com/truenas/apps/discussions) for questions.
4. CI on the PR re-runs render/deploy/health on amd64 + arm64 → another reason
   the image must be multi-arch and run as 568.
5. After merge, Lantern appears in **Discover** on the next catalog sync.

## Key risks / open questions

1. **`BASE_URL` chicken-and-egg (highest UX risk).** QR codes bake in
   `BASE_URL`, which must be the host's LAN IP + the *published* port the user
   just chose — TrueNAS can't auto-derive this. If they mismatch, every QR code
   is dead. Mitigations to design: strong field help text with a worked example;
   consider letting the operator set/override `BASE_URL` at runtime in Settings
   (already partly in the app) so a wrong install-time value is fixable without
   reinstall; document prominently in README.
2. **Non-root + Playwright** (prereq 2) — the single biggest engineering item;
   needs a real `--user 568` test run.
3. **Image size** — ~1.5 GB with Chromium; fine but slows first pull.
4. **Maintenance commitment** — community apps are expected to track upstream;
   each Lantern release needs a new image tag + a catalog PR bumping
   `app_version`/`version`. Consider automating the catalog PR from our release
   workflow later.
5. **`min_scale_version`** — confirm the exact minimum (Electric Eel 24.10.x;
   actual-budget uses `24.10.2.2`). Users on 24.04 or earlier (k3s) can't
   install via the new engine.

## Interim / fallback paths (already partly shipped)

- **Custom App / Install via YAML** — works today; documented in
  [`docs/truenas.md`](../docs/truenas.md). Keep as the "before catalog merge"
  story.
- **Self-hosted custom catalog** — we could publish our own catalog repo (same
  layout, any train name) that users add via **Apps → Discover → Add Catalog**
  pointing at our Git URL. Faster than waiting on iX review and fully under our
  control, at the cost of users not getting it from the default catalog. Good
  stepping stone: build the `ix-dev/community/lantern` files once, host them in
  our own catalog repo first, then submit the same files upstream.

## Suggested order of work

1. GitHub Actions → multi-arch image to GHCR with semver tags.
2. Dockerfile: fix Playwright browser path + verify clean run as `--user 568:568`.
3. Author the five app files; validate with `ci.py` against a local clone of
   `truenas/apps`.
4. Prepare icon + screenshots.
5. (Optional) stand up a self-hosted custom catalog to dogfood.
6. Open the community-train PR; iterate on review.
